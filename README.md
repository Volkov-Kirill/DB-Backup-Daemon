# DB-Backup-Daemon

Фоновый сервис (воркер) на C#, предназначенный для автоматического создания резервных копий баз данных. Приложение самостоятельно выполняет резервное копирование, архивирует данные в защищённый ZIP-архив, проверяет корректность созданного архива, сохраняет историю операций и автоматически удаляет устаревшие резервные копии по заданным правилам хранения.

---

## 📖 Описание проекта

Основная задача проекта — автоматизировать процесс резервного копирования данных и минимизировать риск потери информации.

Приложение позволяет:

* Создавать резервные копии баз данных.
* Архивировать резервные копии в ZIP-архив.
* Защищать архивы паролем.
* Проверять целостность созданных архивов.
* Вести историю резервного копирования.
* Хранить информацию о выполненных операциях.
* Автоматически очищать старые резервные копии.
* Выполнять резервное копирование по таймеру в фоновом режиме.
* Просматривать состояние работы через графический интерфейс.

## 🛠️ Конфигурация приложения (.env)

Для работы приложения используется файл `.env`, расположенный в корневой директории проекта.

Пример конфигурации:

```env
BACKUP_DB_PATH=database.db
BACKUP_DIRECTORY=backups
BACKUP_PASSWORD=12345
BACKUP_INTERVAL_MINUTES=1
APP_DATA_PATH=appdata.db
```

### Описание параметров

| Параметр                | Назначение                       |
| ----------------------- | -------------------------------- |
| BACKUP_DB_PATH          | Путь к файлу базы данных         |
| BACKUP_DIRECTORY        | Папка для хранения архивов       |
| BACKUP_PASSWORD         | Пароль для ZIP-архивов           |
| BACKUP_INTERVAL_MINUTES | Интервал запуска фоновой задачи  |
| APP_DATA_PATH           | Путь к локальной базе приложения |

## ⚙️ Что должно делать приложение (ТЗ)

### 1. Работать в фоне

Сервис запускается и работает в системе непрерывно.

* Использует таймер для запуска задач.
* Не требует постоянного участия пользователя.
* Выполняет резервное копирование автоматически.

### 2. Всеядные бэкапы

Архитектура приложения строится через интерфейсы.

Это позволяет в будущем подключать различные СУБД:

* SQLite
* PostgreSQL
* MySQL
* SQL Server
* Oracle

без переписывания основной логики приложения.

### 3. Безопасность

Каждая резервная копия должна:

* Архивироваться в ZIP.
* Защищаться паролем.
* Храниться отдельно от исходной базы данных.

### 4. Контроль

После создания архива приложение обязано:

* Проверить существование файла.
* Проверить размер файла.
* Проверить возможность открытия архива.
* Зафиксировать результат проверки в журнале.

### 5. Умное удаление (Ротация)

Для экономии дискового пространства применяется следующая схема хранения:

* **7** ежедневных бэкапов.
* **4** еженедельных бэкапа.
* **1** ежемесячный бэкап.

Все резервные копии, не попадающие под данные правила хранения, автоматически удаляются.
